# System PRD: Distributed Real-to-Sim Middleware for Robotic Spray Painting

## 1. Project Overview
The objective is to create a system that captures the precise 6DoF motion of a professional painter, trigger actuation, and 3D geometry of the target object to generate UR robot arm trajectories. The system consists of a Node.js Backend, a Web Dashboard Frontend, and a Unity-based Action Node Client.

## 2. Database Schema (MongoDB / Node.js Backend)
The system operates on a multi-tiered hierarchy to manage companies, their experts, and specific painting tasks.

* **Company:** The client or organization that owns the professionals and jobs.
* **Professional:** The skilled technicians.
    * Fields: `name`, `profile_summary` (optional), `company_id`.
* **Job:** A high-level project belonging to a company.
    * Fields: `company_id`, `description`, array of `record_ids` (future recording sessions).
* **Record:** The actual recording session data.
    * Fields:
        * `state`: Current status (e.g., PENDING, SCANNING, ALIGNING, RECORDING, TRAINING, COMPLETED).
        * `professional_id`: The technician performing the task.
        * `subject`: Target object details (type, `3d_model_url` pointing to an AWS S3 bucket).
        * `sensors_used`: Array (e.g., ["Intel RealSense D435i", "HTC Vive Tracker", "Alicat Flow Meter"]).
        * `alignment_metadata`: Crucial pre-recording data. Stores the VLM alignment parameters generated by RealSense and Vive before the actual recording begins.

## 3. System Components & Responsibilities

### Component A: Signaling Backend & Database (Node.js)
* **Tech Stack:** Node.js, Express, MongoDB (Mongoose), AWS SDK (S3).
* **Role:** RESTful API for managing the schemas and generating pre-signed URLs for direct-to-S3 uploads.
* **WebRTC Signaling (`awrtc_signaling`):** We are strictly using the open-source signaling server from `https://github.com/because-why-not/awrtc_signaling` integrated into our backend to handle SDP/ICE exchanges.

### Component B: Dashboard Web Frontend
* **Tech Stack:** React or Next.js, WebRTC Client (compatible with the awrtc signaling protocol).
* **Role:** Operator control center.
* **Features:** * UI to create and manage Professionals, Jobs, and Records.
    * WebRTC P2P connection to the Unity Node for real-time monitoring.
    * Action triggers (e.g., "START_SCAN", "ALIGN_SENSORS", "START_RECORD", "START_TRAINING", "STOP") sent exclusively via **WebRTC Data Channels**.

### Component C: Action Node Client (Unity Digital Twin & ML)
* **Workspace:** Exists in the local directory named `robot_record`. This project currently only contains the `awrtc` asset.
* **Core Tech Stack:** Unity, OpenXR, RealSense SDK, Qwen3-VL (Ollama/Bedrock), Unity WebRTC (`awrtc`).
* **Robotics Tech Stack:** Unity-Robotics-Hub tools, `URDF-Importer`, Unity `ml-agents`.
* **Hardware Ingestion:** Directly ingests HTC Vive 6DoF data, RealSense point clouds, and Alicat Flow Meter data.

* **Workflow Phase 1 (Scanning & VLM Alignment):** Captures 3D scan via RealSense/Vive. Uploads 3D models (`.obj`/`.ply`) to AWS S3. Uses Qwen3-VL (via Ollama or Bedrock) to align OpenXR and RealSense coordinate systems. Saves S3 URL and `alignment_metadata` to the Backend DB.
* **Workflow Phase 2 (Recording Human Demonstrations):** Listens to WebRTC Data Channels for commands. Records the professional's 6DoF motion and Alicat flow meter (trigger) data locally in real-time to create a demonstration dataset.
* **Workflow Phase 3 (Digital Twin Setup):** * Imports official Universal Robots descriptions (`Universal_Robots_ROS2_Description` URDF/Xacro) using the `URDF-Importer`.
    * Sets up the robot in the scene using `Unity-Robotics-Hub` tools.
* **Workflow Phase 4 (Imitation Learning via ML-Agents):** * Utilizes `unity-technologies/ml-agents`. 
    * Uses the recorded 6DoF trajectory and trigger data from Phase 2 as expert demonstrations.
    * Trains the digital twin UR robot arm to perfectly replicate the spray painting task (Imitation Learning/Behavioral Cloning).
* **Workflow Phase 5 (Robot Execution/Communication):**
    * Once the agent is trained and the trajectory is validated in simulation, the Unity client communicates with the physical UR robot (or URSim).
    * Supported protocols: Direct URScript & TCP/IP (referencing `rparak/Unity3D_Robotics_UR` or `Grand-Garage/unity3d_universal_robot`) or ROS-based communication via `ROS-TCP-Connector` and `ROS-TCP-Endpoint`.
    * Updates `Record` state to COMPLETED.
